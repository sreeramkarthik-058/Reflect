# Architectural Decisions

## 2026-04-26 — Supabase key split: publishable on frontend, service role on backend

**Decision:** Frontend uses `VITE_SUPABASE_PUBLISHABLE_KEY` (`sb_publishable_` format) for auth and read queries. Backend uses `SUPABASE_SERVICE_ROLE_KEY` (`sb_secret_` format) for all entry inserts, bypassing RLS.

**Reason:** The service role key grants full database access and must never be exposed in client-side code. Splitting keys means the frontend can authenticate users while the backend owns all write operations.

**Alternatives considered:** Having the frontend insert directly with the user JWT — rejected because it required complex RLS debugging and left the door open for client-side manipulation of `user_id`.

---

## 2026-04-26 — Explicit Postgres GRANT to authenticated, anon, and service_role

**Decision:** After enabling RLS on `entries`, explicit `GRANT SELECT, INSERT, UPDATE, DELETE ON public.entries TO authenticated, anon, service_role` was required. Same for `GRANT USAGE ON SCHEMA public`.

**Reason:** Supabase RLS policies control row visibility but do not grant table-level access. Without the GRANT, even the service role received `42501 permission denied for table entries`. This is distinct from an RLS policy failure.

**Alternatives considered:** Disabling RLS — rejected; RLS is the correct long-term access model for per-user data.

---

## 2026-04-26 — AI responses generated by backend, not frontend

**Decision:** `POST /api/entries` on the Express backend handles the Claude API call. The frontend never calls the Anthropic API directly.

**Reason:** Keeps `ANTHROPIC_API_KEY` server-side only. Prevents key exposure in browser network tabs, bundled JS, or client-side environment variables.

**Alternatives considered:** Frontend calling Claude directly via `VITE_ANTHROPIC_API_KEY` — rejected immediately; API keys in Vite env vars are visible in the compiled bundle.

---

## 2026-04-26 — Entry save flow: frontend → /api/entries → Supabase + Claude → response

**Decision:** Single `POST /api/entries` call does three things atomically from the frontend's perspective: (1) inserts the entry via service role key, (2) calls Claude Haiku with the entry content, (3) patches `ai_response` on the row and returns the full entry. Frontend shows a loading state during the entire round-trip.

**Reason:** Keeps the frontend simple — one request, one response, one state transition. The backend can fail gracefully on the Claude call (AI error is non-fatal; entry is still saved and returned without `ai_response`).

**Alternatives considered:** Two separate frontend calls (save first, then poll for AI response) — rejected as unnecessarily complex for Phase 2.

---

## 2026-04-26 — Tailwind CSS v4 (CSS-first, no tailwind.config.js)

**Decision:** Project uses Tailwind v4. Design tokens are defined as CSS custom properties in the global stylesheet, not in `tailwind.config.js`. There is no config file.

**Reason:** Tailwind v4 moved to a CSS-first configuration model. Token values (colours, fonts) live in `@theme` blocks in CSS. All utility classes referencing these tokens (e.g. `bg-bg`, `text-gold`, `border-border`) are defined there.

**Alternatives considered:** Staying on Tailwind v3 with a config file — not applicable; project was initialised on v4.
